### 2. Micro XRCE-DDS Agent Docker Integration
**What:** Added Micro XRCE-DDS Agent as a system-wide tool to the Docker image.

**Why:** This is the middleware that bridges PX4 (client) and ROS2 (via DDS). It's the core communication backbone for the entire pipeline.

**Implementation:**
- Added v2.4.3 as a git submodule (same as PX4-Autopilot approach)
- Updated Dockerfile to build and install it as a system tool during image creation
- Uses cmake/make build system (standard C++ workflow)

**Technical detail:** The Dockerfile copies the submodule, builds it in a temp directory, installs binaries to `/usr/local/bin`, updates the library cache with `ldconfig`, then cleans up the temp folder to keep image size reasonable.

COPY src/Micro-XRCE-DDS-Agent ./microxrce-agent-src

## Learning Summary

**Key Concepts Clarified:**

1. **Git flags and build tools:**
   - `-b <branch>`: Checks out a specific branch
   - `cmake`: C++ build configuration (generates Makefiles)
   - `make`: Compiles source code to binaries
   - `make install`: Places binaries on system paths
   - `ldconfig`: Updates library cache so programs can find newly installed libraries

2. **Submodules and Docker workflow:**
   - Submodules are git pointers to external repos, not copies
   - Build folders created during Docker image build don't get committed
   - Full paths (`/home/ws/...`) work from anywhere; relative paths only work from their base directory

3. **Python package management:**
   - `pip install --user` installs to user home (~/.local), not system-wide
   - `pip install <package>==<version>` pins to exact version

4. **Workflow insights:**
   - Clone/download first, verify progress with `ps aux | grep git` and `du -sh`, then let it finish
   - SSHFS is slow; always work inside the devcontainer for large operations
   - Unstaging files: `git restore --staged <file>`
